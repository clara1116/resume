## 1. 产品愿景

### 1.1 一句话描述

**帮助中国互联网实习生将"打杂"经历转化为有说服力的简历故事，避免简历被秒刷。**

### 1.2 核心问题

中国互联网实习生面临独特的简历困境：

- **经历碎片化：** 3 个月实习中做了很多小事，不知道如何整合成体系
- **量化困难：** "优化了接口性能" 但不知道怎么说清楚效果
- **表达空洞：** 简历充斥着"负责XX"、"参与XX"等空泛动词
- **中英文差异：** 无法直接套用英文简历的写作逻辑
- **ATS 焦虑：** 不知道简历能否通过大厂的自动筛选系统

### 1.3 价值主张

| 传统方式 | Next Internship Copilot |
|---------|------------------------|
| 自己回忆 → 遗漏关键经历 | ERF 四阶段访谈 → 系统性挖掘 |
| "负责XX项目" → 毫无说服力 | STAR 重构 → "主导XX优化，P99延迟降低40%" |
| 主观美化 → 面试被戳穿 | Source Trace → 每条陈述可追溯 |
| 简历模板 → 千篇一律 | AI 增强 → 保留个人特色，增强表达力 |
| 隐私裸奔 → 信息泄漏 | 隐私保护层 → 自动检测 + 脱敏 |

## 2. 用户画像

### 2.1 核心用户 Persona

**小陈，21岁，大三**

- 学校：某 211 计算机相关专业
- 实习：正在某中厂做后端开发实习生（第 2 个月）
- 目标：秋招拿到大厂 offer
- 痛点：
  - 每天做的是 CRUD 和修 Bug，不知道怎么写进简历
  - 看到别人的简历有很多量化数据，但自己不知道怎么算
  - 担心简历被机器刷掉，连面试机会都没有
- 行为习惯：
  - 习惯微信沟通，不喜欢打电话
  - 手机重度用户，但简历这种"正式文档"会在电脑上完成
  - 愿意为"能帮我拿到面试"的服务付费（心理价位 30-100 元）

### 2.2 次要用户

**小林，24岁，研二**

- 有多段实习经历，但每段都不长（2-3个月）
- 需要把多段短实习串联成有逻辑的职业叙事
- 关注国际化：准备同时投递国内外公司

### 2.3 用户场景

| 场景 | 使用模式 | 频率 |
|------|---------|------|
| 正在实习中，想持续记录 | Mode A — Weekly Reflection | 每周 1 次 |
| 实习刚结束，急需投简历 | Mode B — Quick Export | 一次性 |
| 已有旧简历，想优化 | Mode B — Quick Export | 一次性 |
| 拿到面试通知，临时改简历 | Dashboard → 快速调整 | 按需 |

## 4. Mode A — Weekly Reflection

### 4.1 设计理念

**为什么是 Weekly 而不是 Daily？**

v2.0 的设计是 Daily Log（每日记录）。经过架构级 Review，改为 Weekly Reflection，理由：

- **降低用户负担：** 实习生每天已经很累，每天写工作日志是额外负担
- **更宏观的视角：** 一周的反思更容易识别"我真正做了什么有价值的事"
- **避免琐碎化：** 每日记录会产生大量"今天修了一个 Bug"之类的琐碎条目
- **符合反思周期：** 一周是一个自然的回顾节奏

### 4.2 用户流程

```
每周固定时间（如周日晚）：
  │
  ▼
打开 Next Internship Copilot → 新建本周反思
  │
  ▼
┌─────────────────────────────────────────────┐
│  Step 1: 自由书写（5分钟）                    │
│  "这周做了什么？想到什么写什么"               │
│  支持 Markdown 快捷输入                       │
│  提示：列出 3-5 件你觉得最重要的事             │
└─────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────┐
│  Step 2: AI 追问（交互式）                    │
│  ERF 四阶段自动进行                           │
│  AI 逐阶段提问，用户回答                       │
│  每阶段结束时展示结构化提取结果                │
│  用户可修改、补充、跳过                        │
└─────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────┐
│  Step 3: 生成素材                            │
│  AI 将对话转化为 3-8 条 STAR 素材            │
│  每条素材标注来源、置信度                      │
│  用户逐条审核：确认 / 修改 / 标记待核实        │
└─────────────────────────────────────────────┘
  │
  ▼
素材存入素材库 → 累积多周后 → 一键生成完整简历
```

### 4.3 进度可视化

Dashboard 展示本周反思状态：

```
上周: ✓ 完成  生成了 5 条素材
本周: ○ 进行中 (阶段 2/4)
```

### 4.4 设计决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 反思频率 | 每周 | 降低认知负担，避免碎片化 |
| 写入方式 | 自由文本 + AI 追问 | 先发散再收敛，符合思维习惯 |
| 可修改性 | 所有 AI 输出可编辑 | 用户掌握最终控制权 |
| 历史回顾 | 按周展示 | 形成成长轨迹，增强成就感 |

## 5. Mode B — Quick Export

### 5.1 设计理念

满足"我现在就需要一份简历"的紧急需求。与 Mode A 的区别：

| 维度 | Mode A | Mode B |
|------|--------|--------|
| 使用场景 | 实习中持续记录 | 急需导出简历 |
| 时间跨度 | 数周 | 一次性 |
| AI 交互 | 深度访谈（4阶段） | 轻量增强（1轮） |
| 输入方式 | 自由反思 | 结构化表单 |
| 输出质量 | 高（多轮追问） | 中（单轮增强） |
| 素材沉淀 | 自动入库 | 可选保存 |

### 5.2 用户流程

```
┌─────────────────────────────────────────────┐
│  Step 1: 填写基础信息                        │
│  ─────────────────────                       │
│  公司名称、实习岗位、起止时间                  │
│  实习内容：自由文本输入                        │
│                                             │
│  实时 PII 扫描提示：                         │
│  ⚠ 检测到手机号，建议修改                     │
└─────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────┐
│  Step 2: AI 增强                             │
│  ────────────────                            │
│  AI 识别可量化的内容 → 追问 2-3 个问题        │
│  生成增强版描述 → 左右对比展示                │
│                                             │
│  原始: "优化了数据库查询"                      │
│  增强: "通过添加联合索引，将订单查询接口的      │
│         P99 延迟从 1.2s 降至 120ms"           │
│         [来源: 用户提到"之前很慢，加了索引"]    │
│                                             │
│  用户: [确认] [修改] [保持原样]               │
└─────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────┐
│  Step 3: 选择模板 → 预览 → 导出              │
└─────────────────────────────────────────────┘
```

### 5.3 Quick Export 特有的设计考量

- **不强制 ERF 全流程：** 用户只想快点导出，不要强迫走完所有阶段
- **但保留追问入口：** 对于明显可以深挖的内容，提供"深入挖掘"按钮，让用户主动选择进入 ERF 访谈
- **最少点击：** 从打开到导出，理想路径三步搞定

## 6. ERF 经验重构框架

### 6.1 框架概览

ERF（Experience Reconstruction Framework）是本产品的核心 AI 方法论，将「模糊的实习记忆」转化为「结构化的简历素材」。

```
Timeline ──▶ People ──▶ Task ──▶ Impact
 (时间线)     (人物)      (任务)     (影响)

  识别关键    还原协作    拆解动作    量化结果
  事件节点    关系深度    技术细节    业务价值
```

### 6.2 四阶段详解

#### 阶段 1: Timeline（时间线梳理）

**目标：** 把散乱的记忆整理成有序的时间节点

**AI 做的事情：**
- 从用户的自由文本中提取事件
- 按时间排序
- 识别"关键节点"（值得深入挖掘的事件）
- 标注置信度

**用户看到什么：**
- "你本周做了这些事情，我标记了 3 个值得深入聊的"
- 可以用拖拽调整顺序、增删节点

#### 阶段 2: People（人物协作）

**目标：** 还原协作方式，体现沟通和推动能力

**AI 做的事情：**
- 识别协作角色（mentor、PM、同事、跨部门）
- 判断协作深度（独立执行 / 协作 / 主导推进）
- 追问"你是怎么说服对方的""如何解决分歧的"

**为什么重要：**
- 大厂筛简历非常看重"推动力"和"跨部门协调"能力
- 实习生最容易忽视的就是"协作"的经历
- 一个好的协作故事比"独立完成任务"更有说服力

#### 阶段 3: Task（任务拆解）

**目标：** 将泛泛的"参与"拆解为具体的技术动作

**AI 做的事情：**
- 拆解任务为动作链（需求 → 设计 → 实现 → 测试 → 上线）
- 识别技术决策点（为什么选这个方案而不是那个）
- 挖掘困难与解决方案

**输出示例：**
```
输入: "参与了下单模块的重构"
拆解:
  1. 梳理现有业务逻辑，画出 15 个接口的调用关系图
  2. 发现 3 处循环依赖，提出拆分方案
  3. 主导接口定义（API Design），编写接口文档
  4. 实现 8 个核心接口，代码量 3000+ 行
  5. 编写 40+ 个单元测试，覆盖率 78%
```

#### 阶段 4: Impact（影响量化）

**目标：** 用数据说话，但不编造数据

**AI 做的事情：**
- 识别可量化的维度（时间、规模、效率、质量）
- 追问基线数据（"优化前是多少？"）
- 辅助推算（"根据你的描述，大约是..."）
- 标注数据来源和置信度

**量化分类：**
| 置信度 | 标注 | 示例 |
|--------|------|------|
| 精确 | ✓ | "接口响应时间从 2300ms 降至 800ms"（用户提供了精确监控数据）|
| 估算 | ~ | "接口响应时间从 ~2s 降至 <1s"（用户凭记忆给了范围）|
| 推算 | ? | "估算每周节省约 5 人天"（AI 基于用户描述推算）|

### 6.3 ERF 的设计原则

1. **渐进深入：** 先宽后深，先易后难，不在一开始就问最难的问题
2. **用户主导：** AI 追问但不替代用户思考，每个阶段都设验证节点
3. **可跳过：** 用户对某个阶段没内容可说时可以跳过
4. **可回溯：** 用户可以随时回到之前的阶段修改

## 7. 隐私保护层

### 7.1 设计哲学

> 简历是高度敏感的个人文档。隐私保护不是可选项，是产品的基础设施。

### 7.2 保护层次

```
Layer 1: 前端实时提示
  - 用户在输入框中输入手机号 → 即时黄框提示
  - 不阻止输入，但提醒风险

Layer 2: AI 深度检测
  - LLM 上下文感知检测（不只是正则）
  - 识别"我家住在北京市朝阳区XX小区3号楼" → 地址
  - 识别"我身份证号是..." → 身份证号

Layer 3: 自动脱敏
  - 简历生成时自动替换
  - 13812345678 → 138****5678
  - 保留脱敏记录，用户可回查

Layer 4: 用户最终审核
  - 导出前展示完整的 PII 扫描报告
  - "以下信息将被脱敏处理，确认？"
  - 用户可逐个撤销脱敏（如果确认无风险）
```

### 7.3 PII 检测类型

| 类型 | 检测方式 | 处理 |
|------|---------|------|
| 手机号 | 正则 + AI | 部分脱敏 |
| 身份证号 | 正则 + 校验位 | 全部删除 |
| 邮箱 | 正则 | 用户确认保留或删除 |
| 银行卡号 | 正则 + Luhn | 全部删除 |
| 家庭地址 | AI 上下文检测 | 脱敏到区级 |
| 内部 URL/IP | 正则 | 删除或替换 |
| 真实姓名 | AI 上下文检测 | 保留（简历需要） |

### 7.4 数据存储策略

- 手机号：仅存储 SHA-256 Hash + AES-256 密文
- 简历内容：明文存储（用户需要编辑），但传输加密
- 验证码：5 分钟过期，用完即删
- DOCX 文件：7 天后自动删除
- 账号删除：软删除 30 天后物理清除所有数据

## 8. 产品原则与边界

### 8.1 产品原则

1. **真实性优先：** AI 增强但不虚构。每一条修改都可以回溯到用户的原始输入。
2. **用户掌控：** AI 是建议者，最终决策权永远在用户手中。
3. **隐私第一：** 敏感信息检测和脱敏贯穿全流程，不做"先收集再处理"。
4. **渐进式复杂度：** Mode B 三步就能导出，Mode A 提供深度体验。用户根据自己的需求选择深度。
5. **中文原生：** 不照搬英文简历的产品逻辑。中文简历有自己的格式、习惯、审美。
6. **不做面试辅助：** MVP 阶段聚焦简历生成，不扩展到面试准备（mock interview 是另一个产品）。

### 8.2 明确不做的事（MVP）

- ❌ AI 模拟面试 / 面试题库
- ❌ 多岗位简历变体（如"投字节和投腾讯用不同简历"）
- ❌ 团队协作（"mentor 帮我看简历"）
- ❌ 简历评分 / ATS 通过率预测
- ❌ LinkedIn/脉脉 导入
- ❌ PDF 生成（仅 DOCX）
- ❌ 移动端 App
- ❌ 英文简历生成
- ❌ 薪资谈判辅助
- ❌ 行业对标分析

这些功能在 MVP 验证后根据用户反馈决策。

### 8.3 竞品分析（中国市场）

| 产品 | 模式 | 优势 | 劣势 |
|------|------|------|------|
| 超级简历 | 模板 + 手动填写 | 知名度高 | 无 AI 深度参与 |
| 知页简历 | 模板市场 | 模板多 | 无经历挖掘能力 |
| 职徒简历 | 在线编辑器 | 编辑器体验好 | 缺乏 AI 增强 |
| ChatGPT 通用 | Prompt 对话 | 灵活 | 无结构化流程，易产生幻觉 |

**我们的差异化：**
- 专为实习生设计（而非通用简历工具）
- ERF 访谈式挖掘（而非"粘贴文本 AI 改写"）
- 隐私保护和反幻觉机制内置（而非靠 Prompt 约束）
